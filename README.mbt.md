# mbt-embed

Tiny MoonBit-native CLI that recursively walks a folder and emits a set of
`.mbt` files exposing every file's bytes through a `Map[String, Bytes]`.

## Layout

```
src/mbt-embed/       the CLI tool (native backend)
example/assets/      sample files to embed
example/consume/     example package that reads embedded bytes
```

## Why a tool, not `include_bytes!`?

MoonBit has no file-include macro yet, so the tool reads files at *build time*
and emits a `.mbt` with the bytes baked in as a literal â€” a compact string
literal for well-formed UTF-8 content, or a `Bytes::from_array([...])` hex
literal otherwise. See "Generated file shape" below.

## Build the generator

The generator uses `moonbitlang/x/fs`, which only provides native/js stubs, so
the generator itself is built on the **native** backend:

```
moon build --target native src/mbt-embed
```

## Run it

```
moon run --target native src/mbt-embed -- <input_dir> <output_dir> [--package <name>]
```

* `<input_dir>`  â€” folder to embed recursively (key = path relative to it).
* `<output_dir>` â€” existing directory the generated `.mbt` files are written into.
* `--package <name>` â€” emitted as a comment only; MoonBit does not need a
  per-file `package` directive, but it documents where the files belong.

One file per embedded entry (`embed_0.mbt`, `embed_1.mbt`, ...) keeps any
single generated file from growing with the number or size of embedded
assets, plus a main `embed.mbt` holding `entries` and `get`, plus a
`moon.pkg` manifest (empty `import {}`, since the generated code never
reaches outside the builtin prelude) so `<output_dir>` is a standalone
MoonBit package on its own. All files must live together in the same
MoonBit package: the per-entry `_embed_N` functions have no `pub` and are
only reachable from `embed.mbt` because MoonBit visibility is per-package,
not per-file.

Empty files are supported (the entry is `_utf8_encode("")`).

## Wiring it into a build

`moon` has no `pre-build` script hook, so do one of the following:

* run `moon run` with the args above from a wrapper shell script / Makefile
  target before `moon build`, or
* run the generator once and check in (or gitignore) the generated file.

## Generated file shape

`embed.mbt`:

```moonbit
// Generated by mbt-embed. DO NOT EDIT.
pub let entries : Map[String, Bytes] = Map::from_array([
  ("hello.txt", _embed_0()),
  ("sub/note.txt", _embed_1()),
])

///|
pub fn get(path : String) -> Bytes? { entries.get(path) }

///|
fn _utf8_encode(s : String) -> Bytes { ... }
```

`_utf8_encode` is only emitted when at least one entry needs it. It re-encodes
a MoonBit `String` (UTF-16 internally) back to UTF-8 bytes by hand, so the
generated output never needs an import beyond the implicit builtin prelude.

`embed_0.mbt` and `embed_1.mbt` (one such file per entry) â€” the content of
`hello.txt` happens to be well-formed UTF-8, so it takes the compact string
form; `sub/note.txt` doesn't (e.g. it's a binary asset), so it falls back to
the hex byte-array form:

```moonbit
// Generated by mbt-embed. DO NOT EDIT.

///|
fn _embed_0() -> Bytes { _utf8_encode("hi there\n") }
```

```moonbit
// Generated by mbt-embed. DO NOT EDIT.

///|
fn _embed_1() -> Bytes { Bytes::from_array([0x68, 0x69, ...]) }
```

Which form a given file gets is decided purely by whether its bytes are
well-formed UTF-8 â€” there's no extension whitelist. For UTF-8-heavy formats
(JSON, SVG, HTML, JS, CSS, plain text, ...) the string form usually costs
about one source byte per embedded byte, versus roughly six for the hex form
(`0xNN, `); genuinely binary files still get the hex form.
